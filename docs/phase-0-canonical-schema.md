# Phase 0 Canonical Schema

## Purpose

This document defines the canonical schema shapes for the first two foundational model families:

- operational transaction header / line
- GL header / line

These are not meant to become one giant polymorphic production table. They are the reference contract that concrete business tables should follow.

Examples:

- `Invoice` should follow the transaction header contract
- `InvoiceLineItem` should follow the transaction line contract
- `JournalEntry` / future posting batches should follow the GL header contract
- `JournalEntryLineItem` / future posted lines should follow the GL line contract

The goal is to make future modules consistent, auditable, and performant.

## Design Rules

1. Use concrete tables per business object, not one universal transaction table.
2. Keep business transactions separate from accounting postings.
3. Make source lineage mandatory.
4. Make indexing part of the design.
5. Support dimensions, custom fields, open items, FX, and intercompany at the contract level.
6. Model journal adjustments and journal reversals so they are easy to reconcile and can participate cleanly in open-item management where applicable.

## Canonical GL Header Contract

### Purpose

Represents one posting event to the ledger.

### Journal adjustment and reversal principle

Manual and system-generated journal adjustments should be modeled so they help account reconciliation rather than obscure it.

That means journal architecture should support:

- explicit source and purpose classification
- clear reversal linkage
- open-item participation where the journal is intended to create, settle, adjust, or reopen an open item
- drill-through from reconciliation and roll-forward views back to the originating journal and line
- predictable activity typing for adjustment, reversal, accrual, deferral, reclass, and settlement-related journal activity

### Required identity fields

- `id`
- `journalNumber`
- `postingType`
- `postingStatus`

### Required date and period fields

- `postingDate`
- `accountingPeriodId`
- `fiscalYear`
- `fiscalPeriod`

### Required scope fields

- `subsidiaryId`
- `bookId` where multi-book is later enabled

### Required source lineage

- `sourceType`
- `sourceId`
- `sourceNumber`
- `sourceRunId`
- `reversesGlHeaderId`
- `reversalReasonCode`

### Required workflow / control fields

- `isReversal`
- `isAutoGenerated`
- `isIntercompany`
- `isOpenItemRelevant`

### Required narrative / metadata fields

- `memo`
- `reference`

### Required audit fields

- `createdAt`
- `updatedAt`
- `createdById`
- `postedById`
- `reversedById`

### Recommended indexes

- unique index on `journalNumber`
- index on `postingDate`
- index on `accountingPeriodId`
- index on `subsidiaryId`
- index on `sourceType`
- index on `sourceId`
- index on `sourceRunId`
- index on `reversesGlHeaderId`
- compound index on `(subsidiaryId, postingDate)`
- compound index on `(sourceType, sourceId)`

## Canonical GL Line Contract

### Purpose

Represents the debits and credits that hit the ledger.

### Reconciliation-friendly journal expectation

Journal lines should make reconciliation easy by preserving enough structured context to answer:

- what account movement is this line supporting?
- is it opening, operational, settlement, accrual, reversal, reclass, or FX-related?
- is it linked to an open item?
- is it reversing a prior line or offsetting one?
- what reconciliation or roll-forward bucket should it land in?

### Required identity and linkage

- `id`
- `glHeaderId`
- `lineNumber`
- `reversesGlLineId`

### Required accounting fields

- `accountId`
- `debitAmount`
- `creditAmount`
- `netAmount`

### Required currency fields

- `transactionCurrencyId`
- `localCurrencyId`
- `functionalCurrencyId`
- `transactionAmount`
- `localAmount`
- `functionalAmount`
- `exchangeRateId` or equivalent FX source reference

### Required dimensional fields

- `classId`
- `locationId`
- `projectId`
- custom dimension references

### Required movement / analytical fields

- `activityType`
- `openItemId`
- `counterpartyType`
- `counterpartyId`
- `reconciliationGroupKey`

### Required source lineage

- `sourceTransactionId`
- `sourceTransactionLineId`
- `sourceScheduleLineId`
- `sourceApplicationId`
- `sourceRunDetailId`
- `reconciliationInstanceId`

### Required intercompany fields where relevant

- `intercompanyPairKey`

- `dueToFromIndicator`

### Required audit fields

- `createdAt`
- `updatedAt`

### Recommended indexes

- index on `glHeaderId`

- index on `openItemId`

- compound index on `(accountId, postingDate)` through join or denormalized strategy where reporting requires it
- compound index on `(accountId, subsidiaryId, accountingPeriodId)` through join or denormalized strategy where reporting requires it

## Extension Hooks

The canonical schema must be compatible with:

- custom field values on headers and lines
- custom dimensions on transactions and GL
- open-item applications
- schedule-line posting
- run-generated transactions
- integration-generated transactions

## Retrofit Guidance Against Current App

Existing models do not need immediate destructive replacement.

Instead, use this document as the target contract for:

1. retrofitting current transaction families
2. shaping all new transaction families
3. designing shared posting services
4. deciding where to normalize current posting data further

Near-term retrofit candidates:

- `Invoice` / `InvoiceLineItem`
- `Bill` / `BillLineItem`
- `CashReceipt` / `CashReceiptApplication`
- `BillPayment` / `BillPaymentApplication`
- `JournalEntry` / `JournalEntryLineItem`
