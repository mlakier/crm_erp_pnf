# Phase 0 Activity Type Catalog

## Purpose

This document defines the canonical activity-type catalog used to classify accounting movements across:

- GL lines
- roll-forward reporting
- open-item events
- clearing events
- FX remeasurement and translation
- intercompany activity
- reporting and KPI packages

It builds on:

- [phase-0-architecture-blueprint.md](./phase-0-architecture-blueprint.md)
- [phase-0-canonical-schema.md](./phase-0-canonical-schema.md)
- [phase-0-open-item-schema.md](./phase-0-open-item-schema.md)
- [phase-0-clearing-schema.md](./phase-0-clearing-schema.md)
- [phase-0-fx-intercompany-schema.md](./phase-0-fx-intercompany-schema.md)

The goal is to ensure accounting movement classification is explicit, consistent, and reusable across posting, roll-forward, reporting, and close processes.

## Design Goals

1. Make accounting movement types explicit rather than inferred from transaction names or account numbers.
2. Support balance-sheet roll-forwards with reliable movement buckets.
3. Support drill-through from summary movement categories to underlying GL and source transactions.
4. Support specialized accounting flows such as clearing, realized FX, unrealized FX, CTA, and intercompany.
5. Keep the catalog stable enough for reporting while still extensible for future modules.

## Core Rule

Every accounting-relevant GL line should carry an explicit activity type.

Where a single source transaction can create multiple kinds of accounting movement, each resulting GL line should carry the activity type that best describes its accounting purpose.

## Canonical Structure

## 1. `ActivityTypeDefinition`

### Purpose

Defines one allowed accounting movement category.

### Recommended canonical fields

- `id`
- `code`
- `name`
- `category`
- `status`
- `description`
- `defaultRollForwardGroup`
- `isSystemDefined`
- `isOpenItemRelevant`
- `isClearingRelevant`
- `isFxRelevant`
- `isIntercompanyRelevant`
- `createdAt`
- `updatedAt`

### Notes

- system-defined activity types should exist for the core accounting engine
- custom additions may be allowed later, but the base catalog should be strongly governed

## 2. `ActivityTypeAssignmentRule` (optional, later)

### Purpose

Supports defaulting or validation logic that maps source posting scenarios to activity types.

### Recommended canonical fields

- `id`
- `sourceType`
- `sourceSubType`
- `transactionType`
- `glAccountType`
- `postingSide`
- `activityTypeId`
- `priority`
- `status`
- `createdAt`
- `updatedAt`

### Notes

- this can come later if needed
- early implementation may use hardcoded posting-engine rules while preserving the canonical catalog

## Canonical Activity Categories

The catalog should group activity types into broader reporting categories.

Suggested top-level categories:

- opening
- operational
- settlement
- accrual_deferral
- schedule_based
- fx
- intercompany
- adjustment
- close
- statistical if later needed

## Core System Activity Types

Below is the recommended first-wave system catalog.

## Opening and Close

- `opening_balance`
  opening balances loaded or carried forward into a reporting period

- `closing_carryforward`
  period-end close/carry-forward activity

## Operational

- `operational_movement`
  ordinary business posting movement not better classified elsewhere

- `inventory_movement`
  inventory valuation movement

- `fixed_asset_activity`
  fixed asset acquisition, capitalization, or base asset movement

- `depreciation`
  depreciation expense and accumulated depreciation movement

## Settlement and Open Item

- `settlement_application`
  clearing/application of an open item

- `writeoff`
  write-off activity tied to settlement or collection/payable resolution

- `reopen`
  reopening an item previously considered closed

- `refund_settlement`
  refund-related settlement movement

## Accrual / Deferral / Schedule

- `accrual`
  accrual creation or adjustment

- `reversal`
  reversal of prior accruals or other entries where reversal is the accounting purpose

- `deferral`
  deferred revenue or deferred expense recognition into a balance-sheet deferral account

- `revenue_recognition`
  revenue recognition event generated by rev rec schedule/runs

- `amortization`
  amortization event generated by amortization schedules/runs

- `prepaid_release`
  prepaid expense release from asset to expense

## FX

- `realized_fx`
  realized gain/loss created by settlement/clearing

- `unrealized_fx`
  unrealized remeasurement gain/loss

- `translation`
  translated balance movement

- `cta`
  cumulative translation adjustment

## Intercompany

- `intercompany_settlement`
  due-to / due-from settlement

- `intercompany_reclass`
  intercompany reclassification or balancing movement

- `elimination`
  consolidation elimination movement

## Adjustment

- `manual_adjustment`
  manual accounting adjustment

- `reclassification`
  balance reclass not better captured by another activity type

- `allocation`
  allocation-generated movement

- `fmv_allocation`
  fair market value allocation movement

## Optional Future Activity Types

These may become relevant later as modules expand:

- `lease_activity`
- `tax_adjustment`
- `budget_only`
- `statistical`
- `manufacturing_variance`
- `landed_cost`

## Roll-Forward Principle

Roll-forward reporting should rely primarily on activity types and secondarily on account and dimensions.

### Why

Without explicit activity types, reports tend to depend on:

- transaction labels
- ad hoc account mapping
- brittle source-type assumptions

That becomes unreliable once the platform supports:

- rev rec
- amortization
- FX
- intercompany
- allocations
- automation runs

### Reporting expectation

Roll-forwards should be able to group by:

- activity category
- activity type
- account
- subsidiary
- dimension
- period

with drill-through to:

- GL header / line
- source transaction
- source run

## Relationship to GL

### Rule

`GlLine` should carry:

- `activityTypeId`
or an equivalent stable activity-type reference

### Expectation

Every posting engine should assign activity type intentionally.

Examples:

- invoice revenue line may be `operational_movement`
- clearing-created realized FX line should be `realized_fx`
- month-end remeasurement line should be `unrealized_fx`
- rev rec run output line should be `revenue_recognition`
- amort run output line should be `amortization`

## Relationship to Open Items and Clearing

### Open items

Open-item entry types and activity types are related but not identical.

For example:

- an `OpenItemEntry` may be of type `application`
- the corresponding GL line activity type may be `settlement_application`

### Clearing

Clearing documents should drive activity types such as:

- `settlement_application`
- `refund_settlement`
- `writeoff`
- `realized_fx`

## Relationship to FX and Intercompany

FX and intercompany require explicit movement classification.

Expected activity types include:

- `realized_fx`
- `unrealized_fx`
- `translation`
- `cta`
- `intercompany_settlement`
- `intercompany_reclass`
- `elimination`

These should not be inferred loosely from source transaction names.

## Relationship to Reporting and KPI Packages

The financial package and KPI layer should use activity types to:

- explain movement changes period over period
- distinguish operating results from FX or close adjustments
- support board and management reporting
- support deep drill-through on balance-sheet movements

## Governance Principle

The first-wave activity-type catalog should be centrally governed.

That means:

- core system types should not be casually renamed
- reporting should use stable codes
- new types should be added deliberately and documented

## Indexing Guidance

Where activity type is stored on GL and other reporting-heavy tables, index it where helpful for:

- roll-forward reporting
- FX reporting
- close reporting
- management package reporting

At minimum, expect indexes on:

- `activityTypeId`
- `accountId + activityTypeId`
- `subsidiaryId + activityTypeId`
- `postingDate + activityTypeId`

depending on the concrete GL implementation.

## Frontend Contract

The shared frontend should expose activity typing through:

- GL impact detail
- roll-forward reports
- financial package drill-through
- exception and close reporting
- analytics filters

Users should be able to see the movement classification without needing to infer it manually.

## Retrofit Guidance for Current App

Current and near-term transaction families that should adopt the catalog early:

- journals
- invoices
- bills
- invoice receipts
- bill payments
- customer refunds
- FX remeasurement outputs
- intercompany journals

These are the most important first-wave adopters because they underpin the accounting control layer.
