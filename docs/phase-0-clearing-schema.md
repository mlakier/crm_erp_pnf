# Phase 0 Clearing Schema

## Purpose

This document defines the canonical architecture for:

- clearing document headers
- clearing document lines
- automated settlement lineage
- realized FX attribution at settlement
- reversal and exception handling for clearing activity

It builds on:

- [phase-0-architecture-blueprint.md](./phase-0-architecture-blueprint.md)
- [phase-0-canonical-schema.md](./phase-0-canonical-schema.md)
- [phase-0-open-item-schema.md](./phase-0-open-item-schema.md)

The goal is to make clearing a first-class accounting model family rather than an implicit side effect of applications, payments, or receipts.

## Design Goals

1. Represent clearing as an auditable accounting event.
2. Support automatic clearing generation wherever the source workflow is deterministic.
3. Support one-to-one, one-to-many, and many-to-many settlement relationships.
4. Attribute realized FX to the exact settlement event and line(s) that caused it.
5. Preserve reversal and historical lineage without overwriting prior clearing history.
6. Support both operational drill-through and financial reporting.

## Scope

This model applies to any settlement or clearing event, including:

- invoice receipt clears invoice(s)
- bill payment clears bill(s)
- credit memo clears invoice(s)
- vendor credit clears bill(s)
- customer refund consumes unapplied cash or credit balance
- deposit-related settlement where applicable
- intercompany settlement
- manual settlement between open items
- future automated bank matching outputs

It does not require all of those to be implemented immediately, but the model must support them.

## Core Rule

A clearing event is not just an amount reduction on an open item.

A clearing event is:

- a distinct settlement document
- plus one or more line-level clearing relationships
- plus linked open-item applications and GL results
- plus realized FX where applicable

## Canonical Tables

## 1. `ClearingDocumentHeader`

### Purpose

Represents one settlement / clearing event.

Examples:

- one cash receipt clears one or more invoices
- one bill payment clears one or more bills
- one customer refund consumes a prior customer credit balance
- one manual settlement clears two historical open items

### Recommended canonical fields

- `id`
- `clearingNumber`
- `clearingType`
- `status`
- `subsidiaryId`
- `transactionCurrencyId`
- `localCurrencyId`
- `functionalCurrencyId`
- `clearingDate`
- `postingDate`
- `accountingPeriodId`
- `sourceTransactionType`
- `sourceTransactionId`
- `sourceRunId`
- `counterpartyType`
- `counterpartyId`
- `transactionAmount`
- `localAmount`
- `functionalAmount`
- `realizedFxLocalAmount`
- `realizedFxFunctionalAmount`
- `memo`
- `reversesClearingDocumentId`
- `reversedByClearingDocumentId`
- `autoGenerated`
- `automationSource`
- `exceptionStatus`
- `createdAt`
- `updatedAt`
- `createdById`
- `postedById`
- `reversedById`

### Suggested clearing types

- receipt_application
- payment_application
- credit_memo_application
- vendor_credit_application
- refund_settlement
- bank_match_settlement
- intercompany_settlement
- manual_settlement
- writeoff_settlement

### Suggested status values

- draft
- pending_review
- posted
- reversed
- void

### Notes

- `autoGenerated` indicates whether the platform created the clearing event automatically.
- `automationSource` can record the originating workflow, such as `invoice_receipt_post`, `bill_payment_post`, or `bank_match_run`.
- `exceptionStatus` is for exception-oriented automation handling, not the normal accounting lifecycle.

- `realizedFxFunctionalAmount`

- Realized FX can exist at the line level even when the header also carries rolled-up realized FX totals.

## 3. Relationship to `OpenItemApplication`

### Rule

`OpenItemApplication` and `ClearingDocument` are related but not interchangeable.

Recommended relationship:

- `ClearingDocumentHeader` = the accounting settlement event
- `ClearingDocumentLine` = each clearing component
- `OpenItemApplication` = the explicit open-item settlement relationship

### Practical guidance

For standard receipt/payment/credit settlements:

- create one `ClearingDocumentHeader`
- create one or more `ClearingDocumentLine`
- create one or more `OpenItemApplication`
- create one or more `OpenItemEntry`

All of them should reference each other where useful for lineage.

## 4. Relationship to GL

### Rule

Clearing documents are not the GL itself, but they must tie directly to the posting outcome.

### Expectations

- every posted clearing document should be traceable to one or more `GlHeader` / `GlLine` records
- realized FX postings should be attributable to clearing documents and lines
- reversing a clearing event should create reversing GL, not mutate prior posted lines

### Typical GL impact examples

- clear receivable/payable balance
- recognize realized FX gain/loss
- record write-off if part of the settlement

## Lifecycle Rules

### Creation

When a deterministic source workflow settles open items, create the clearing document automatically.

Examples:

- posting an invoice receipt with applications
- posting a bill payment with applications
- applying a vendor credit
- applying a credit memo

### Manual creation

Manual clearing remains supported for exceptional or controlled accounting cases, but should not be the default operating model.

### Posting

When the clearing document posts:

- create linked `OpenItemApplication` rows
- append linked `OpenItemEntry` rows
- create or link the resulting `GlHeader` / `GlLine` rows

### Reversal

Reversal must preserve history:

- create a reversing clearing document or reversing clearing lines
- create reversing open-item application / entry records where applicable
- create reversing GL where applicable

Do not overwrite the original settlement record.

## Automation Principle

Clearing should be automated wherever practical.

Preferred standard behavior:

- operational flow determines settlement relationship
- platform generates clearing document automatically
- platform posts open-item and GL consequences automatically
- user reviews only exceptions

### Automation-friendly flows

- invoice receipt applications
- bill payment applications
- customer refund consumption of prior balances
- vendor credit and credit memo application
- deterministic bank matching
- deterministic intercompany settlement runs

### Exception cases

The platform should flag for review when:

- multiple plausible matches exist
- cross-currency rules are ambiguous
- open-item balances do not reconcile cleanly
- intercompany counterpart cannot be validated
- required rate context is missing

## Realized FX Principle

Realized FX should be recognized through clearing, not inferred loosely after the fact.

### Why

Realized FX occurs because:

- the original open item was established at one rate context
- the settlement occurs at another rate context

The clearing document is the natural accounting record that bridges those two contexts.

### Design expectation

The model must support:

- original rate context reference
- settlement rate context reference
- line-level realized FX amounts
- rolled-up header realized FX totals
- linked GL lines that post realized gain/loss

## Reporting Expectations

The model should support:

- clearing history by customer/vendor/employee/intercompany
- realized FX by clearing event
- manual vs automated clearing reporting
- exception reporting for failed or ambiguous clearing
- reversal reporting
- drill-through from GL to clearing to source transaction

## Frontend Contract

The shared frontend should expose clearing through:

- applications / allocations sections
- detail-page related records
- historical clearing reports
- realized FX drill-through
- exception queues for uncertain automated clearing
- bank matching and settlement review workflows

## Retrofit Guidance for Current App

The current app already has transaction-specific application records such as:

- receipt applications
- bill payment applications

These should evolve toward the canonical model by:

1. preserving existing operational behavior first
2. introducing canonical clearing records behind the scenes
3. linking applications to clearing headers/lines
4. linking realized FX posting behavior to the clearing family
